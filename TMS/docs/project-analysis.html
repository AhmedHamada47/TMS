1. Project Overview

TaskFlow (TMS — Task Management System) is an ASP.NET Core 9.0 MVC web application for managing personal tasks with multi-user support. It uses SQLite via Entity Framework Core 9.0 for data storage, custom cookie authentication (no ASP.NET Core Identity), and BCrypt.Net-Next for password hashing.

11. Problems & Recommended Fixes

🔴 Critical / High Priority

1. Dashboard shows global stats, not user-scoped

Controllers/HomeController.cs:21-38 — All 11 queries run against the entire Tasks/Users tables without UserId filter. A user sees task counts, user counts, and category data from all users.

Fix: Add .Where(t => t.UserId == currentUserId) to all task queries. For user-level stats, keep the global count as the "Total Users" metric but scope task stats to the current user.

2. No CSRF on Logout POST

While Logout has [ValidateAntiForgeryToken] in the controller, the sidebar renders a logout form. If the token is valid, this is fine — but confirm the form tag helper generates the token.

Fix: Already using asp-antiforgery="true" by default in form tag helpers. Verify manually. If missing, add @Html.AntiForgeryToken() or ensure asp-controller/asp-action tag helpers are used.

3. No HTTPS enforcement in production

app.UseHttpsRedirection() is called but there's no HSTS (UseHsts) configured. In production, passwords travel over the network.

Fix: Add app.UseHsts() in non-development environments. Configure HSTS max-age and includeSubDomains in production.

4. Category Name unique index is global, not per-user

Data/AppDbContext.cs:39 — entity.HasIndex(c => c.Name).IsUnique(). This means User A cannot create a "Development" category if User B already has one (though due to ownership checks, they wouldn't see it, but the DB will reject the insert).

Fix: Make the unique index composite: HasIndex(c => new { c.Name, c.UserId }).IsUnique(). Also need a new migration.

5. No input sanitization on task/category names

User-submitted strings (Title, Description, Category Name) are stored as-is and rendered with @Model.Title (HTML-encoded by Razor by default). However, no server-side sanitization for XSS exists.

Fix: Razor's @ syntax auto-encodes, but for defense-in-depth, validate input with a whitelist approach or use System.Web.Security.AntiXss. At minimum, ensure no raw HTML rendering (@Html.Raw) on user content.

🟡 Medium Priority

6. Lockout is per-email but stored in memory (not persisted)

Controllers/AccountController.cs:14 — ConcurrentDictionary resets on app restart. A sophisticated attacker could time the restart window.

Fix: Persist failed attempts in the database (User.LockoutEnd, User.AccessFailedCount columns) or use a distributed cache (Redis) for production.

7. No audit logging

No logging of who created/updated/deleted tasks or categories. UpdatedAt tracks the date but not UpdatedBy.

Fix: Add UpdatedByUserId to TaskItem. Add an AuditLog table or Serilog structured logging for all data mutations.

8. Password strength validation is duplicated

Rules (8+ chars, letter, digit) are checked in Register POST, ChangePassword POST, and reflected in ChangePasswordViewModel via [StringLength(200, MinimumLength = 8)]. The letter/digit check is only in the controller, not in the ViewModel attributes.

Fix: Create a custom [PasswordStrength] validation attribute to DRY the regex checks.

9. DbContext concurrency risk

Dashboard runs 11 parallel queries via Task.WhenAll. While EF Core supports multiple queries in parallel on the same context, the default SQLite provider does not support multiple concurrent queries — it will throw InvalidOperationException.

Fix: Enable Pooling=true in the connection string OR remove Task.WhenAll and run queries sequentially. The queries are fast (simple COUNTs) so sequential is acceptable for SQLite.

10. Profile/Index loads all user tasks into memory

Controllers/ProfileController.cs:26 — .Include(u => u.Tasks) loads ALL tasks for rendering stats. For users with thousands of tasks, this is slow.

Fix: Replace with aggregate queries: _context.Tasks.Where(t => t.UserId == userId).GroupBy(...).Select(...) to compute stats at the DB level.

🟢 Low Priority / Polish

11. Hardcoded avatar URLs

Avatar URLs point to a specific CDN (pub-a981f7fafe3c46e98d60519aae806cf8.r2.dev). If the CDN goes down, avatars break with no fallback.

Fix: Add a DefaultAvatarUrl fallback in config or use initials-based avatar generation (e.g., ui-avatars.com API).

12. jQuery dependency for validation only

The _ValidationScriptsPartial loads jQuery 3.7.1 + jQuery Validate + Unobtrusive just for client-side validation. This is ~87KB of JS.

Fix: Migrate to native HTML5 validation with data-val-* attributes and a lightweight custom validator, removing jQuery dependency.

13. No pagination on Categories Index

Categories are loaded all at once. Fine for small datasets, but could be slow with 100+ categories.

Fix: Add pagination similar to Tasks Index if users can have many categories.

14. No confirmation dialog for task Delete (inline)

Views/Tasks/Index.cshtml:133 — Inline delete uses return confirm(). This is fine but could be more user-friendly with a modal.

Fix: Replace the inline confirm() with a Bootstrap modal for consistency with the dedicated Delete page.

15. Chart.js loaded on every page (not just dashboard/profile)

Chart.js 4.4.7 (~200KB) is loaded in _Layout.cshtml via CDN, even on pages that don't use charts (Tasks Index, Categories, etc.).

Fix: Move Chart.js script to @section Scripts in only the views that need it (Home/Index and Profile/Index).

16. var in inline scripts

Views/Home/Index.cshtml:289, Views/Account/Login.cshtml:14 — Inline script blocks use var instead of let/const. The Register.cshtml also uses var in its avatar picker script.

Fix: Replace var with let/const in all inline scripts for consistency with site.js.

17. Old _Sidebar.cshtml still in project

The file Views/Shared/_Sidebar.cshtml is replaced by the ViewComponent but still present. It's not referenced but creates confusion.

Fix: Delete _Sidebar.cshtml (confirmed not referenced by Layout).

18. No favicon

No <link rel="icon"> in any layout. Browsers will show a generic icon.

Fix: Add a favicon.ico or SVG favicon in wwwroot/ and reference in _Layout.cshtml and auth page layouts.
